Блок 1. Модель данных (SQLite + EF Core)
Сущность Employee: ФИО, email, даты создания/обновления
Сущность Survey: название, описание, статус (Черновик / Активен / Завершён), email создателя, даты
Сущность Question: текст, тип (текст / один вариант), обязательность, порядок
Сущность QuestionOption (для вариантов ответа)
Сущность SurveyAssignment: кто оценивает (evaluatee) + кто отвечает (respondent) — матрица
Сущность RespondentLink: token, respondentId, surveyId, expiresAt (14 дней), isCompleted
Сущность Answer + AnswerItem (по каждому вопросу и каждому оцениваемому)
Миграции EF Core, seed: 5–10 тестовых сотрудников + 1 шаблон опроса
Политика хранения: ответы живут 1 месяц (поле или фоновая задача — зафиксировать в коде/README)
Критерий готовности: схема покрывает матрицу N×M, одна ссылка на респондента, ответы привязаны к паре «респондент → оцениваемый».

Блок 2. Справочник сотрудников
API: CRUD сотрудников (GET /employees?search=)
Поиск по ФИО и email (autocomplete)
UI: список + форма добавления/редактирования
Валидация email, уникальность email
Компонент EmployeePicker — переиспользовать в матрице и создании опроса
Критерий готовности: в UI можно найти человека по имени и выбрать из базы.

Блок 3. Создание опроса и шаблоны
API: создать / редактировать / удалить опрос (только в статусе «Черновик»)
API: вопросы — добавить / изменить / удалить / сортировка
UI: список опросов со статусами
UI: конструктор анкеты (текст + выбор одного варианта, обязательный флаг)
Шаблон по умолчанию — при создании подставлять пример вопросов (акцент на текстовые, не баллы)
Поле email создателя опроса (куда слать итоги)
Сохранение черновика без запуска
Критерий готовности: создатель собирает анкету из текстовых вопросов, видит список своих опросов.

Блок 4. Матрица «кто кого оценивает» ⭐
API: сохранить / получить матрицу для опроса
UI: столбцы = оцениваемые, строки = респонденты (или наоборот — как договоритесь, заказчик просил столбцы)
Чекбоксы / переключатели: кто кого оценивает
Выбор людей из справочника (EmployeePicker)
Минимум 1 респондент на оцениваемого; максимум — вся команда
Редактирование только до запуска (статус «Активен» — блокировка)
Превью: сколько ссылок / писем уйдёт при запуске
Критерий готовности: для команды из 5 человек можно за 2 минуты расставить, кто кого оценивает.

Блок 5. Персональные ссылки и email ⭐⭐ (P0)
API: запуск опроса — генерация уникальных token-ссылок на каждого респондента
Срок жизни ссылки: 14 дней (expiresAt)
Интеграция отправки email (SMTP / MailKit / dev-режим с логом в консоль)
Шаблон письма: ФИО, список «кого оцениваете», кнопка-ссылка
UI: кнопка «Запустить и разослать» + статус отправки по каждому
Повторная отправка одному респонденту (если не прошёл)
Dev-fallback: показать ссылки списком, если SMTP недоступен
Критерий готовности: респондент получает письмо, открывает ссылку, видит своих оцениваемых.

Блок 6. Прохождение опроса (страница респондента) ⭐⭐ (P0)
Страница /survey/:token — без логина
Заголовок: кто вы и кого оцениваете (это главное в UI)
Если несколько оцениваемых — по очереди (wizard / шаги)
Форма ответов: текстовые поля крупно, варианты — вторично
Валидация обязательных вопросов
API: сохранение ответов (транзакция на оцениваемого)
Повторный вход по той же ссылке → не форма, а просмотр своих ответов (read-only)
Истёкшая ссылка → понятное сообщение
Подтверждение «Ответы сохранены»
Критерий готовности: полный цикл: ссылка → ответы → повтор → только просмотр.

Блок 7. Результаты для создателя опроса
API: агрегация ответов по опросу / по оцениваемому
UI: страница результатов (создатель знает id опроса или секретную ссылку — без auth, обсудить: ссылка-админка или email-only)
Группировка: оцениваемый → респондент → ответы (не анонимно)
Прогресс: кто уже ответил / кто нет
Email создателю: «опрос завершён» или «все ответили» + ссылка на результаты
Кнопка «Завершить опрос» (статус → «Завершён»)
Критерий готовности: создатель видит все ответы всех респондентов по каждому оцениваемому.

Блок 8. Экспорт в Word (.docx) — P3
Библиотека генерации docx на .NET (DocumentFormat.OpenXml / DocX)
Структура документа: ФИО оцениваемого → блок респондент + все ответы → следующий
API: GET /surveys/{id}/export.docx
UI: кнопка «Скачать Word»
Отправка docx на почту создателю (опционально)
Критерий готовности: один файл со всеми ответами, удобно копировать/архивировать.

Блок 9. UI/UX и стиль Directum
Общий layout: шапка, подвал, цвета Directum
Логотип, шрифты (или близкие системные)
Акцент на удобство ввода текста (большие textarea, понятные подписи)
Пустые состояния, ошибки, loading
Десктоп 1280+ (мобилка — не в MVP)
Критерий готовности: визуально лучше старого сайта, заказчик не ругается на «говно-UI».
